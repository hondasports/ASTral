# Stuck Advisor

## 自動トリガー

- 同じtestまたはcommandが2回以上、同じ理由で失敗した。
- 同じreview findingで2回以上差し戻された。
- OXC、SQLite、watcher、MCPの同じ前提を変えずに修正を繰り返している。

## 状況整理

推測を分離し、まず初稿を作る。

```text
目的:
正確な症状・エラー:
試したことと結果:
最後に成功した状態:
確認済みの事実:
未確認事項:
固定していた前提:
```

## 仮説の初稿

1. 独立した仮説を3つ作る。
2. 1つは問題定義そのものを疑う。
3. 各仮説へ根拠、最小検証、棄却条件を書く。
4. 同じ修正案の言い換えを別仮説として数えない。

## `grill-me`による洗練

次の場合は、次のアクションを選ぶ前に仮説初稿を`grill-me`へ渡す。

- 技術調査だけでは選べず、要件、互換性、許容精度、運用優先順位へ依存する。
- 「正しい挙動」の定義自体が複数ある。
- performanceとcorrectness、鮮度と処理負荷などの優先順位が未定。
- 失敗を回避する変更が公開契約やdata formatを変える。
- 3仮説が同程度に妥当で、最初に検証すべき案を決められない。

### 洗練手順

1. コード、ログ、test、ADRから答えられる事項を先に埋める。
2. ユーザー判断が必要な分岐だけを`grill-me`へ渡す。
3. 一度に1問だけ聞き、推奨回答と理由を添える。
4. `GRILL RESULT`を仮説の前提、優先順位、棄却条件へ反映する。
5. 洗練後の仮説から、最も安全で速い検証を1つだけ選ぶ。

利用中のエージェントがSkill chainingに対応しない場合は、ユーザーへ`/grill-me`の実行を依頼する。

ユーザー判断が不要で、最小検証が一意なら、壁打ち省略理由を一行記録して進める。

## 次のアクション

- 5分程度で結果が分かる、変更量の少ない検証を優先する。
- 外れた仮説を同じ形で繰り返さない。
- 大規模refactorやschema変更は、局所検証を先に行う。

## 出力

```text
状況整理:
仮説初稿:
- 仮説A:
- 仮説B:
- 仮説C（問題定義の見直し）:
GRILL RESULT（実施時）:
壁打ちで変更した前提・優先順位:
壁打ち省略理由（省略時）:
洗練後の仮説:
選んだ次の1アクション:
停止・escalate条件:
```

3仮説を棄却しても進展がない場合は、追加情報と未確認範囲を示してescalateする。
